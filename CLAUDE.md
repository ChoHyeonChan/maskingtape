## 2. 🚨 대회 규정 — 위반 시 실격/수상취소. 예외 없음

1. **라이선스는 Apache-2.0 하나로 통일.** 모든 새 파일은 이 라이선스로 작성. 다른 라이선스의 코드를 복사해 붙이지 마라 (스택오버플로/블로그 코드 포함 — AI에게 "처음부터 작성"을 시켜라).
2. **의존성 추가 전 라이선스 확인 필수.** 허용: MIT, Apache-2.0, BSD, ISC. **금지: GPL·AGPL·SSPL·비상업 전용 라이선스**(GPL 금지는 대회 규정이 아닌 팀 방침 — Apache-2.0 통일 유지 목적). 라이브러리를 하나라도 추가하면 저장소의 `SBOM.md` 표에 즉시 한 줄 추가(이름/버전/라이선스/공식 저장소 URL/사용 목적). 심사에 **라이선스 검증 단계가 별도로 존재**하고, SBOM은 결과보고서 붙임1로 제출된다.
3. **제품 코드에서 외부 상용 AI API 호출 금지.** OpenAI·Anthropic·Google 등 클라우드 LLM API를 부르는 코드를 작성하지 마라. LLM은 오직 **로컬 Ollama(오픈웨이트 모델: Qwen 계열 우선 — Apache-2.0이라 서류가 깔끔)**. ※ "개발 도구로 AI를 쓰는 것"은 허용 — 제품이 API를 호출하면 안 된다는 뜻이다.
4. **AI가 짠 코드는 이해한 사람만 머지.** 발표평가에서 "이 코드 동작 원리"를 답 못 하면 감점(규정 명시). 습관: 코드 생성 후 AI에게 **"방금 코드의 동작 원리를 3줄로 설명해줘"** → 그 설명을 PR 본문에 붙인다.
5. **진짜 개인정보를 저장소에 절대 넣지 마라.** 테스트·예제 데이터는 전부 합성(가짜)으로 생성. 실제 이름·전화번호·주민번호가 커밋되는 순간 "개인정보 보호 도구가 개인정보를 유출"한 게 된다. AI Hub 등 외부 데이터셋 원본도 재배포 제한 때문에 커밋 금지(로컬에서만 사용).
6. **다른 프로젝트 코드 이식 금지.** 팀원이 다른 해커톤·과제·대회에서 만든 코드를 이 저장소로 가져오지 마라. 모든 코드는 이 대회 기간에 새로 작성한다 (중복수혜·모방 규정 대응).
7. **커밋은 반드시 본인 GitHub 계정으로.** 개인별 기여도가 심사에 반영된다 — 커밋 이력이 곧 증거다.

## 3. 코드 구조 — 처음부터 세분화, 나중에 리팩토링 없음

저장소 구조(스켈레톤)가 절대 기준이다. **자기 파트 폴더 밖은 수정하지 않는다** (공용 수정 필요 시 팀장에게).

```
packages/core/            # Python 엔진 — 순수 로직만, UI/네트워크 코드 금지
  maskingtape/
    types.py              #   Detection 공용 타입
    pipeline.py           #   조립만 담당 (로직 없음)
    cli.py                #   명령줄 도구
    detectors/            #   탐지기 1종 = 파일 1개 (rrn.py가 참고 구현 — 이 패턴을 따른다)
    anonymizers/          #   마스킹 전략 1종 = 파일 1개 (mask.py가 참고 구현)
  tests/                  #   합성 데이터로만 테스트
packages/mcp-server/      # MCP 서버 — core를 import해서 노출만
apps/api/                 # FastAPI — core 래핑, 웹·데스크톱 공용 백엔드
apps/web/                 # 웹 플레이그라운드 (프론트)
apps/desktop/             # Flutter 데스크톱 앱
bench/                    # 합성 데이터 생성기 + 평가 스크립트 + 리포트
```

- **파일 하나 = 역할 하나. 300줄 넘으면 분리를 검토한다.** "일단 한 파일에 넣고 나중에 정리"는 금지 — 그 "나중"은 오지 않는다.
- 의존 방향은 단방향: `apps → packages/core`. core는 외부 패키지 최소, apps를 절대 import하지 않는다.
- UI / 로직 / 데이터 / 설정은 항상 파일 단위로 분리한다.

## 4. 바이브 코딩 규칙 (AI에게 시킬 때)

1. 세션 시작: **이 파일 + 자기 파트 폴더의 README**를 AI에게 먼저 읽힌다.
2. **작게 시켜라.** "화면 전체 만들어줘"(✗) → "탐지 결과 하이라이트 컴포넌트 하나 만들어줘"(✓). 큰 요청은 구조를 망가뜨린다.
3. 머지 전 3종 세트: ① 직접 실행해서 눈으로 확인 ② 테스트 통과 ③ AI가 써준 "동작 원리 3줄" PR에 첨부.
4. PR 승인은 코드 리뷰어 2인(코드 리딩 가능한 팀원)만 한다. 리뷰어도 이해 안 되면 승인하지 말고 질문.
5. AI가 새 라이브러리를 마음대로 추가하면 §2-2 절차(라이선스 확인 + SBOM)부터. AI는 이 규칙을 모르니 네가 지켜줘야 한다.
