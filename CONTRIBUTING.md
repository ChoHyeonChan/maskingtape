# 기여 가이드 (팀 협업 규칙)

> **팀원에게**: AI(Claude Code, Cursor, ChatGPT 등)로 작업한다면 **이 파일과 [CLAUDE.md](CLAUDE.md)를 먼저 읽히고** "이 규칙대로 작업해줘"라고 지시하세요. 이 문서 하나면 우리 협업 방식이 그대로 굴러갑니다.

## 왜 이렇게 하나 — 깃 사용 흔적이 곧 점수다

2026 오픈소스 개발자대회 심사 기준에 **깃 관리체계가 직접 배점**돼 있습니다.

| 평가 | 항목 | 배점 | 요구 |
|---|---|---|---|
| 1차 서면 | 프로젝트 팀워크 | 6점 | ※ github **Issues, review, pull requests, commit, merge, 커뮤니티** 등 관리체계 |
| 2차 발표 | 커뮤니티 확장 가능성 | 10점 | 품질관리·개발 방법론·**개발 로드맵 등의 관리체계**, 커뮤니티 참여와 지적 자산 공유 |

즉 **코드만 좋으면 안 되고, "어떻게 협업했는지"가 저장소에 남아야** 합니다. 그래서 아래 흐름을 지킵니다.

---

## 작업 흐름 — 이슈 → 브랜치 → PR → 머지

### 1단계. 이슈부터 만든다 (코드보다 먼저)

작업을 시작하기 전에 **무엇을 왜 하는지** 이슈로 남깁니다. 큰 작업은 잘게 쪼개세요.

- 제목: `[파트] 할 일` 예) `[API] POST /scan 엔드포인트 구현`
- **라벨**을 답니다: `part: core` / `part: api` / `part: web` / `part: desktop` / `part: bench` / `part: mcp` / `part: repo` (+ `bug`, `enhancement`, `documentation`, `security`)
- **마일스톤**을 답니다: `Phase 1 — 기능 완성` / `Phase 2 — 통합·배포` / `Phase 3 — 제출물`
- 본문에 배경·할 일 체크리스트·완료 기준을 씁니다.

### 2단계. 그 이슈에서 브랜치를 판다

**GitHub 이슈 페이지 오른쪽 `Development` → `Create a branch`** 를 누르면 이슈에 연결된 브랜치가 자동으로 만들어집니다. 이렇게 하면 이슈↔브랜치↔PR↔커밋이 한 줄로 엮여 보입니다.

```bash
git fetch origin
git switch <생성된-브랜치-이름>
```

수동으로 만들 때는 이슈 번호를 앞에 붙입니다: `26-scan-endpoint`

> **`main`에서 직접 작업하지 않습니다.** (보호 설정으로 막혀 있습니다)

### 3단계. 작업하고 커밋한다

- 커밋 메시지는 **무엇을 왜** 바꿨는지 한국어로 적습니다.
- 커밋은 **본인 GitHub 계정**으로 (개인 기여도가 심사에 반영됩니다).
- 새 라이브러리를 추가했다면 **같은 PR에서 [SBOM.md](SBOM.md)에 한 줄 추가**합니다. (라이선스 검증이 2차 심사에 별도로 있습니다)

### 4단계. PR을 올린다 — 본문에 `Closes #이슈번호`

```bash
git push -u origin <브랜치>
gh pr create   # 또는 GitHub 웹에서
```

PR 본문에 **`Closes #26`** 을 반드시 씁니다. 그래야 머지될 때 이슈가 자동으로 닫히고 연결이 기록됩니다.

PR 본문에는 [템플릿](.github/PULL_REQUEST_TEMPLATE.md)대로:
1. **무엇을** 했는지
2. **동작 원리 3줄** — AI가 짠 코드라면 필수입니다. 2차 발표에서 "이 코드 어떻게 동작하죠?"에 답하지 못하면 감점입니다.
3. **검증 근거** — 테스트 결과, 실행 화면 등 *실제로 돌려본 증거*

### 5단계. 리뷰받고 머지한다

- **리뷰는 팀원 누구나** 할 수 있습니다. 승인 1명 + CI 통과가 머지 조건입니다.
- **자기 PR은 자기가 승인할 수 없습니다** (GitHub 정책). 팀 채팅으로 리뷰를 요청하세요.
- 이해가 안 되는 코드는 **승인하지 말고 질문**하세요. 리뷰 코멘트도 심사에서 보는 협업 기록입니다.
- 머지되면 이슈는 자동으로 닫힙니다.

## AI에게 시킬 때

1. 세션 시작 시 **이 파일 + [CLAUDE.md](CLAUDE.md) + 자기 파트 폴더의 README**를 읽힙니다.
2. **작게 시킵니다.** "화면 전체 만들어줘"(✗) → "탐지 결과 하이라이트 컴포넌트 하나"(✓)
3. 머지 전 3종 세트: ① 직접 실행해 눈으로 확인 ② 테스트 통과 ③ 동작 원리 3줄을 PR에 첨부
4. AI는 우리 대회 규칙을 모릅니다. **[CLAUDE.md](CLAUDE.md) §2(실격 방지)를 지키는지 사람이 확인**하세요.

### 복붙용 지시문

```
이 저장소의 CONTRIBUTING.md와 CLAUDE.md를 먼저 읽고, 그 규칙대로 작업해줘.
- 작업 전에 이슈를 만들고(라벨·마일스톤 포함), 그 이슈 번호로 브랜치를 만들어서 작업해
- PR 본문에는 "Closes #이슈번호", 무엇을/동작 원리 3줄/검증 근거를 포함해
- main에 직접 커밋하지 말고, 새 의존성을 추가하면 SBOM.md도 같은 PR에서 갱신해
- 테스트는 반드시 합성(가짜) 데이터로만 작성해
```

## 지켜야 할 선 (어기면 팀 전체 실격)

자세한 내용은 **[CLAUDE.md](CLAUDE.md) §2**에 있습니다. 요약하면:

1. 새 코드는 전부 **Apache-2.0**. 다른 라이선스 코드를 복사해 붙이지 않습니다.
2. 의존성 추가 시 **라이선스 확인 + SBOM 기재**. 허용: MIT·Apache-2.0·BSD·ISC.
3. **제품 코드에서 상용 AI API 호출 금지.** LLM은 로컬 Ollama + 오픈웨이트 모델만.
4. **진짜 개인정보를 저장소에 넣지 않습니다.** 테스트·예시는 전부 합성 데이터.
5. 다른 대회·과제에서 만든 코드를 가져오지 않습니다.

## 🔒 보안이 최우선입니다

이 저장소는 **개인정보를 보호하는 도구**입니다. 여기서 정보가 새면 제품 자체가 부정됩니다.

- **탐지 실패도 유출입니다.** 마스킹이 덜 되면 개인정보가 그대로 남습니다.
- 에러 메시지·로그에 **입력 원문을 넣지 않습니다** (개인정보가 로그로 샙니다).
- 파일을 다루는 코드는 **경로·크기·덮어쓰기를 검사**합니다.
- 정규식 반복에는 **상한**을 둡니다.
- 실패는 **안전한 쪽으로** — 마스킹에서 "덜 가리기"는 유출, "더 가리기"는 안전합니다.
